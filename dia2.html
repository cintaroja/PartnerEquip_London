Observability for Resilience

Ramgopal Rajan - Solutions Architect leader
10:00 - 10:30

How Do We Measure Resilience?

Availability = mean time to failure / (mean time before failure + mean time to recover)

Diagnosing Issues

Dependency Failure

  • Queremos saber el origen del fallo
  • Con el traceID se intentará trazar toda la comunicación
  • X-Ray será el servicio usado

System Component Failure

  • Si un web server is unhealthy puede que tengas un 11% de fallos que te valga pero para el cliente no es válido
  • Lo que necesitas es chequear el health del web server en concreto que se vaya a llamar desde el ALB (instance level)
  • Usaremos dimensionality ("show me [metric] per [dimension]")
    • Metric being latency, errors, requests...
    • Dimension being API, webpage, instanceID
  • Lo haremos con CloudWatch Embedded Metric Format

Ran into a Limit

  • Observe resource utilization
  • Esto trata de resolver de manera proactiva las llegadas a límites
  • Se van a poder crear alarmas asociadas a los thresholds de los límites
  • Si tienes alarmas asociadas a los autoscaling groups cuando escalas tienes que variar la alarma

Traffic Spike

  • Observe traffic volume
  • Cuando tienes un spike de tráfico una de las cosas que quieres es saber la fuente
  • La recomendación es usar CloudWatch Contributor Insights Rules
  • Se va a poder filtrar por top 10 contributors
  • Traffic insights va a permitir unos filtros mucho mayores

Bad Deployment

  • Si aparece un error en un bad deployment las best practices dicen que tienes que empezar con una máquina, luego dos, etc.
  • Se pueden usar alarmas compuestas (por ejemplo, si una medida de CPU sube por sí sola no significa nada pero si lo hace a la vez que la memoria posiblemente sí)

Summary

  • Usar dimensiones para las métricas
  • Logs estructurados (contributor insights, metrics insights, log insights...)

Detecting and Mitigating Gray Failures

Rachna Bhalla
15:15 - 15:45

Differential Observability

Son datos de análisis que no hacen saltar alarmas pero sí que reflejan errores.

Fault Containers

  • Regions
  • Availability Zone
  • Instance
  • Software

Single Instance Gray Failures

Si una instancia falla dentro de un grupo de muchas, puede que la alarma no salte porque el sistema sigue funcionando.

Health Checks

  • Shallow health checks - Detecta en el ELB, y reemplaza la instancia
  • Deep health checks - Detectado en ELB pero no se reemplaza la instancia

Health Checks Not Always Tell the Whole Story

  • Outlier detection
  • Comparing metric data among instances
  • Spotting an outlier -> You do this with lambda (pulling or with an alarm trigger)

Outlier Detection Considerations

  • Hiding failures
  • Detecting false positives

Availability Zone Gray Failures

  • Multiple AZs show impact but 1 is more impacted (outlier)
  • Clear signal 1 AZ is impacted and the others no (outlier or composite alarms)
  • Aggregate signals with cloudwatch composite alarms
  • All fault containers are impacted because of shared resource (composite alarms)

When an AZ Gray Failure Happens You Can:

  • Wait for the impairment
  • Shift away from AZ
  • Failover to another region

Data Plane Controlled Evacuation

  • Amazon Route 53 App Recovery Controller (ARC) zonal shift
  • Self managed HTTP endpoint (example S3)
